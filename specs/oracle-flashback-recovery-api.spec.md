# Oracle Flashback Recovery API Spec

> Generated from: `sop/oracle-flashback-recovery.md`（DBA-SOP-014）
> Generated by: `.claude/skills/sop-to-spec`
> 實作 agent 只需讀本文件，不需回讀 SOP。

服務名稱：`flashback-api`，port **8003**，三層式架構（api / service / repository），
遵循 `docs/architecture/service-layering.md` 的 DI 與測試慣例（wiki-processor 同款：
`core/deps.py` lru_cache providers + `Depends`）。

## 1. Domain Model

| 實體 | 欄位 |
|------|------|
| DatabaseStatus | `log_mode: str`（"ARCHIVELOG"/"NOARCHIVELOG"）、`flashback_on: bool`、`db_state: str`（"OPEN"/"MOUNT"/"READ_ONLY"/"FLASHBACKED"）、`current_scn: int`、`oldest_flashback_scn: int`、`oldest_flashback_time: str`（ISO8601）、`retention_minutes: int`、`fra_limit_bytes: int`、`fra_used_bytes: int`、`estimated_flashback_size: int` |
| RestorePoint | `name: str`（大寫）、`scn: int`、`time: str`、`guarantee: bool`、`storage_size: int` |
| RecycleBinEntry | `owner: str`、`object_name: str`（`BIN$...`）、`original_name: str`、`droptime: str` |
| TableInfo | `owner: str`、`table_name: str`、`row_movement: bool` |
| AuditEntry | 見第 7 節 |

## 2. Endpoints 總表

| Method | Path | 風險等級 | 對應 SOP |
|--------|------|---------|----------|
| GET | `/health` | read | — |
| GET | `/flashback/status` | read | §2（P1–P4 檢查彙總） |
| GET | `/restore_points` | read | §3.2 |
| POST | `/restore_points` | reversible | §3.1 |
| DELETE | `/restore_points/{name}` | reversible | §3.3 |
| GET | `/recyclebin?owner=` | read | §4.2 步驟 1 |
| POST | `/flashback/table` | reversible | §4.1 |
| POST | `/flashback/drop` | reversible | §4.2 |
| POST | `/flashback/database` | **irreversible** | §5 步驟 1–4（至 READ ONLY 驗證） |
| POST | `/flashback/database/finalize` | **irreversible** | §5 步驟 4–5（RESETLOGS） |
| GET | `/audit/log` | read | §8 |

通用規則：
- 所有非 read 端點受 `X-API-Key` 保護（`FLASHBACK_API_KEY` 未設 = dev 模式不驗，
  比照 wiki-processor 的 `PROCESSOR_API_KEY` 模式，request 時讀 env）
- 所有非 read 端點寫 audit（含 dry_run 與失敗的嘗試）
- reversible / irreversible 端點都有 `dry_run: bool = true`——預設只檢查不執行，
  response 回 `checks`（每個前置條件的結果）；`dry_run=false` 才真的執行
- irreversible 端點額外必填：`confirm: str`（固定 token `"I-UNDERSTAND-DATA-LOSS"`）
  與 `approval_id: str`（變更審批單號，非空）。缺任一 → **428** Precondition Required

## 3. 各端點規格

### GET /flashback/status
回 DatabaseStatus 全部欄位 + `fra_usage_percent: float` + `preconditions`：
```json
{
  "log_mode": "ARCHIVELOG", "flashback_on": true, "db_state": "OPEN",
  "current_scn": 2000000,
  "oldest_flashback_scn": 1500000, "oldest_flashback_time": "2026-06-11T09:00:00",
  "retention_minutes": 1440,
  "fra_limit_bytes": 10737418240, "fra_used_bytes": 4294967296, "fra_usage_percent": 40.0,
  "estimated_flashback_size": 2147483648,
  "preconditions": {"P1_archivelog": true, "P2_flashback_on": true, "P3_fra_space": true}
}
```

### GET /restore_points
`{"restore_points": [RestorePoint...]}`，依 scn 排序。

### POST /restore_points
Request：`{"name": str（必填，轉大寫存）, "guarantee": bool = true, "dry_run": bool = true}`

| 前置條件 | 違反 HTTP | error_code |
|---------|----------|-----------|
| P3 FRA 使用率 < 85% | 409 | ORA-19809 |
| 同名 restore point 不存在 | 409 | ORA-38796 |

執行效果：以目前 `current_scn` 建立 restore point。Response：建立後的 RestorePoint。

### DELETE /restore_points/{name}
Query：`dry_run: bool = true`。不存在 → 404 ORA-38780。
Response 含被刪 restore point 的完整資料（guarantee GRP 刪除是收尾動作，SOP §6）。

### GET /recyclebin
Query `owner`（選填，過濾）。`{"entries": [RecycleBinEntry...]}`，droptime 倒序。

### POST /flashback/table
Request：
```json
{"owner": "SCOTT", "table_name": "EMP",
 "target": {"scn": 1700000} 或 {"timestamp": "2026-06-12T09:00:00"},
 "enable_row_movement": false, "dry_run": true}
```
`target` 必須恰好一個 key。

| 前置條件 | 違反 HTTP | error_code |
|---------|----------|-----------|
| 表存在 | 404 | （null） |
| P1 ARCHIVELOG | 409 | （null，message 說明） |
| P4 目標 SCN/時間 ≥ oldest_flashback | 409 | ORA-38729 |
| P6 ROW MOVEMENT 已啟用 | 409 | ORA-08189 |
| 目標 SCN ≤ current_scn | 422 | （null） |

`enable_row_movement=true` 時自動先執行 `ALTER TABLE ... ENABLE ROW MOVEMENT`（消除 P6 違反）。
執行效果：表回溯。**Response 必含 `prior_scn`（操作前 current_scn）**——SOP §4.1 步驟 2
的回退資訊，搞錯可用它再 flashback 回來。

### POST /flashback/drop
Request：`{"owner": str, "table_name": str（original_name）, "rename_to": str|null, "dry_run": true}`

| 前置條件 | 違反 HTTP | error_code |
|---------|----------|-----------|
| P5 recyclebin 啟用且物件在其中 | 404 | ORA-38305 |
| 還原目標名稱不與現有表衝突（衝突且未給 rename_to） | 409 | ORA-38312 |

同名多版本：還原 droptime 最近的一筆（SOP §4.2 注意事項）。
Response：`{"restored_as": str, "from_entry": RecycleBinEntry, "note": "索引仍為 BIN$ 系統名，需手動 rename"}`。

### POST /flashback/database
Request：
```json
{"target": {"restore_point": "BEFORE_UPGRADE_20260611"} 或 {"scn": ...} 或 {"timestamp": "..."},
 "dry_run": true, "confirm": "I-UNDERSTAND-DATA-LOSS", "approval_id": "CHG-2026-0612-001"}
```

| 前置條件 | 違反 HTTP | error_code |
|---------|----------|-----------|
| confirm + approval_id 齊備（dry_run=false 時） | 428 | （null） |
| P1 ARCHIVELOG | 409 | （null） |
| P2 flashback_on | 409 | （null） |
| P3 FRA < 85% | 409 | ORA-19809 |
| P4 目標 ≥ oldest_flashback | 409 | ORA-38729 |
| restore_point 存在（target 用名稱時） | 404 | ORA-38780 |
| db_state 為 OPEN（未在進行中的 flashback） | 409 | （null） |

執行效果（對應 SOP §5 步驟 2–4 前半）：SHUTDOWN IMMEDIATE → STARTUP MOUNT →
FLASHBACK DATABASE → ALTER DATABASE OPEN READ ONLY。結束時 `db_state = "FLASHBACKED"`
（read-only 驗證中）。Response：`{"db_state": "FLASHBACKED", "flashed_back_to_scn": int,
"next_step": "人工驗證資料後呼叫 POST /flashback/database/finalize"}`。

### POST /flashback/database/finalize
Request：`{"dry_run": true, "confirm": "I-UNDERSTAND-DATA-LOSS", "approval_id": str}`

| 前置條件 | 違反 HTTP | error_code |
|---------|----------|-----------|
| confirm + approval_id（dry_run=false 時） | 428 | （null） |
| db_state == "FLASHBACKED" | 409 | （null） |

執行效果（SOP §5 步驟 4 後半–5）：OPEN RESETLOGS，`db_state = "OPEN"`，
current_scn 重置遞增、`resetlogs_time` 記錄。Response 含
`{"warning": "RESETLOGS 完成，舊備份基線失效，立即執行 RMAN 全備（SOP §5 步驟 5）"}`。

### GET /audit/log
`{"entries": [AuditEntry...]}`，時間倒序。Query：`limit: int = 100`。

**OracleRepository 介面**（每方法 docstring 註明對應 SQL，真實實作照抄執行，mock 模擬效果）：

| 方法 | 對應 SQL/操作 |
|------|--------------|
| `get_status()` | `v$database`（log_mode/flashback_on/current_scn）、`v$recovery_file_dest`、`v$flashback_database_log`、`SHOW PARAMETER db_flashback_retention_target` |
| `list_restore_points()` | `SELECT ... FROM v$restore_point ORDER BY scn` |
| `create_restore_point(name, guarantee)` | `CREATE RESTORE POINT <name> [GUARANTEE FLASHBACK DATABASE]` |
| `drop_restore_point(name)` | `DROP RESTORE POINT <name>` |
| `list_recyclebin(owner)` | `SELECT ... FROM dba_recyclebin [WHERE owner=...] ORDER BY droptime DESC` |
| `get_table(owner, table)` | `SELECT row_movement FROM dba_tables WHERE owner=... AND table_name=...` |
| `enable_row_movement(owner, table)` | `ALTER TABLE <o>.<t> ENABLE ROW MOVEMENT` |
| `flashback_table(owner, table, scn)` | `FLASHBACK TABLE <o>.<t> TO SCN <scn>`（timestamp 由 service 換算為 scn 邊界檢查後仍以原語法下達） |
| `flashback_drop(owner, table, rename_to)` | `FLASHBACK TABLE <o>.<t> TO BEFORE DROP [RENAME TO <r>]` |
| `flashback_database(scn 或 restore_point)` | `SHUTDOWN IMMEDIATE; STARTUP MOUNT; FLASHBACK DATABASE TO ...; ALTER DATABASE OPEN READ ONLY` |
| `open_resetlogs()` | `SHUTDOWN IMMEDIATE; STARTUP MOUNT; ALTER DATABASE OPEN RESETLOGS` |
| `append_audit(entry)` / `list_audit(limit)` | 審計存放（real：audit 表；mock：in-memory） |

**Mock 初始狀態**（測試與文件範例都用這份）：

```python

current_scn=2_000_000,
oldest_flashback_scn=1_500_000, oldest_flashback_time="2026-06-11T09:00:00",
retention_minutes=1440,
fra_limit_bytes=10 * 2**30, fra_used_bytes=4 * 2**30,        # 40%
estimated_flashback_size=2 * 2**30,
restore_points={"BEFORE_UPGRADE_20260611": RestorePoint(scn=1_800_000, guarantee=True, ...)},
tables={("SCOTT", "EMP"): row_movement=True, ("SCOTT", "DEPT"): row_movement=False},
recyclebin=[RecycleBinEntry(owner="SCOTT", object_name="BIN$jx8kQ3vT==$0",
                            original_name="BONUS", droptime="2026-06-12T08:30:00")],
audit=[]
```
mock 的每次寫入操作 `current_scn += 1`；`fra_used_bytes` 建 GRP 時 +1GiB（可測 P3）。

## 6. 錯誤模型

統一 error body（自訂 exception handler）：
```json
{"detail": "human readable message", "error_code": "ORA-38729"}
```

| error_code | 對應前置條件 | HTTP | 處置建議（寫進 detail） |
|-----------|-------------|------|------------------------|
| ORA-38729 | P4 超出保留範圍 | 409 | 改用 RMAN point-in-time recovery |
| ORA-08189 | P6 無 ROW MOVEMENT | 409 | 帶 enable_row_movement=true 重試 |
| ORA-38305 | P5 不在 recyclebin | 404 | 改用 RMAN 或輔助庫匯出 |
| ORA-19809 | P3 FRA 不足 | 409 | 加大 FRA、刪過期備份/已用 GRP |
| ORA-38780 | restore point 不存在 | 404 | 查 GET /restore_points |
| ORA-38796 | restore point 同名 | 409 | 改名或先 DELETE |
| ORA-38312 | 還原目標名稱衝突 | 409 | 帶 rename_to 重試 |

## 7. 審計（AuditEntry）

所有非 read 端點，**成功、失敗、dry_run 都寫**：

```json
{"timestamp": "ISO8601", "operator": "api-key 前 8 碼或 'dev'",
 "operation": "flashback_table", "target": "SCOTT.EMP",
 "target_scn": 1700000, "target_time": null,
 "dry_run": false, "approval_id": null,
 "result": "success" | "rejected:ORA-08189" | "error:<msg>"}
```

## 8. 測試計畫（MOCK_ORACLE=true）

| 群組 | 案例 |
|------|------|
| status | 回傳全部欄位、preconditions 旗標正確 |
| restore_points | list；create dry_run 不改狀態；create 真執行；同名 409 ORA-38796；FRA 撐爆後 create 409 ORA-19809；delete；delete 不存在 404 ORA-38780 |
| flashback/table | dry_run 回 checks；真執行回 prior_scn 且 scn 前進；DEPT 無 row movement 409 ORA-08189；enable_row_movement=true 後成功；目標 < oldest 409 ORA-38729；表不存在 404；target 同時給 scn+timestamp 422 |
| flashback/drop | 還原 BONUS 成功（從 recyclebin 消失）；不在 recyclebin 404 ORA-38305；同名衝突 409 ORA-38312；rename_to 成功 |
| flashback/database | dry_run 回 checks 不改狀態；缺 confirm 428；缺 approval_id 428；restore point 不存在 404；成功後 db_state=FLASHBACKED；db 非 OPEN 時再呼叫 409 |
| finalize | db_state 非 FLASHBACKED 409；成功後 OPEN + warning |
| auth | FLASHBACK_API_KEY 設定時，mutation 無 key 401；read 不需 key |
| audit | 上述操作（含 dry_run 與 rejected）都留下 entry |

## 9. Out of Scope（SOP 中不自動化的步驟）
